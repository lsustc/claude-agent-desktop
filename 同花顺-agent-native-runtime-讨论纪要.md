# 同花顺炒股客户端与 Agent 平台融合讨论纪要

日期：2026-04-01

## 讨论背景

本轮讨论的起点是：如何把传统的同花顺炒股客户端和 OpenClaw 这类 Agent 平台融合。

讨论过程中，问题定义发生了几次修正：

1. 最初的旧框架是“给炒股客户端加一个 AI 助手或聊天入口”。
2. 结合知识库中的 Agent 相关材料后，判断修正为：重点不是再做一个问财，而是让客户端本身变成 Agent 的运行时。
3. 结合 `场景.md` 后，进一步明确：目标不是“AI 对话产品”，而是“用户炒股的 runtime”。
4. 最终关键问题被收敛为：

`怎么把炒股客户端做成一个 agent-native runtime。`

并且已经确认：

`有没有问财，其实无所谓。`

## 一、关键判断的演进

### 第一阶段：从“AI 功能增强”到“Agent 平台化”

最初的一个基础判断是：

- 不能只是给老客户端接一个聊天框
- 也不能只做一层“问答 + 资讯总结”
- 更有价值的方向是把行情、资讯、选股、盯盘、持仓、交易前后辅助等能力工具化，再由 Agent 调用

这个阶段的中心思想是：

`不是给老客户端加 AI，而是把老客户端升级成 Agent 原生的金融终端。`

### 第二阶段：从“问财能力延伸”到“运行时层重构”

用户指出“这些事问财 AI 已经做了”，因此判断被修正。

修正后的框架是：

- 问财已经占据了“答案层”
- OpenClaw 这类平台代表的是“运行时层、编排层、记忆层、多 Agent 层”
- 同花顺客户端真正稀缺的不是再做一遍分析，而是账户上下文、交易上下文、终端体验、风控边界和长期运行能力

因此新的判断是：

`问财负责聪明，同花顺负责在我的账户和终端里持续做事。`

再进一步，问题变成：

- 同花顺是被外部 Agent 集成的底层能力提供方
- 还是自己上升为面向用户的 Agent runtime

### 第三阶段：从“聊天入口”到“生成式金融工作台”

在阅读 `场景.md` 之后，产品定义进一步收敛。

核心不是聊天，而是：

- 页面可以被 Agent 动态生成
- 页面可以保存、复用、继续追问
- Agent 的 thinking 和 tool 调用不应成为主视图
- 真正的主视图应该是 widget、可视化结果和讲解内容

这时的产品定义可以概括成：

`这是一个由实时数据流驱动、由 Agent 生成界面、由 memory 持续个性化的炒股 runtime。`

## 二、基于 `场景.md` 的产品理解

`场景.md` 中定义了五个主场景：

1. 初始化：首页由 memory 和夜间任务结果自动生成
2. 个股技术面讲解：在 K 线和分时图上自动画线、解释
3. 行业分析：自动生成行业 F10、产业链和关联个股
4. 选股：根据自然语言需求筛选股票并生成列表
5. 识图：从照片中识别股票并加入列表

同时，文档对界面风格做了明确取向：

- 不喜欢传统对话列表
- thinking/tool 过程只需用 ghost 样式短暂停留
- widget 和结果内容占据视觉中心
- text 用旁白形式出现
- 用户真正关心的是结果，而不是中间推理链

最重要的一句产品定义来自文档：

`我们提供的是一个用户炒股的 runtime。`

这意味着客户端不再只是固定页面集合，而是：

- 一个用户可持续使用的工作台
- 一个 Agent 可生成页面和功能的运行时
- 一个可沉淀 memory、可保存 artifacts、可反复复用的环境

## 三、最终收敛的问题定义

经过几轮修正后，真正的问题不再是：

- 怎么把同花顺和 OpenClaw 接一下
- 怎么给炒股软件加一个对话框
- 怎么复刻问财已有能力

而是：

`怎么把炒股客户端做成一个 agent-native runtime。`

这个 runtime 的硬定义是：

`客户端里的任何核心能力，都必须既能被人操作，也能被 Agent 以结构化方式调用；任何重要页面，都不是写死页面，而是“数据 + 意图 + 组件树”的运行结果。`

## 四、agent-native runtime 的核心架构

### 1. 五层基础结构

为了把客户端变成 runtime，需要至少五个基础层：

1. `Data Plane`
   行情、K 线、分时、公告、新闻、财务、F10、自选、图片识别结果等结构化数据层。

2. `Tool Plane`
   讲解 K 线、画线、生成行业链、选股、建列表、对比个股、生成复盘、创建监控等能力的工具层。

3. `Memory Plane`
   用户偏好、常看板块、常用指标、最近浏览、保存的页面、历史分析上下文。

4. `Render Plane`
   Agent 不直接输出长文本或自由 HTML，而是输出 widget spec / page spec，由客户端渲染。

5. `Policy Plane`
   权限、延迟、数据时效性、动作确认、风控边界。

### 2. 快系统和慢系统

`场景.md` 已经点出了一个关键事实：盯盘和分析不是一回事。

因此底层必须拆成两套 loop。

#### 快系统

面向：

- 盯盘
- 异动
- 实时提醒
- 盘口解释
- 高频更新的行情展示

特点：

- 预计算
- 事件驱动
- 固定组件
- 低延迟
- 不依赖 LLM 临场思考

#### 慢系统

面向：

- 复盘
- 行业分析
- 选股
- 诊股
- 识图后的结构化分析
- 页面生成与沉淀

特点：

- Agent 规划
- 工具调用
- 搜索、验证、组织内容
- 生成大 widget 和页面
- 允许较长响应时间

关键结论：

`很多 AI 金融产品做不起来，是因为拿慢系统去顶快系统。`

### 3. 页面不是聊天结果，而是运行结果

客户端不应以气泡对话为主视图，而应把页面视为 Agent 的运行结果。

更稳的方式不是让 Agent 自由生成整页 HTML，而是：

- 客户端内置金融组件系统
- Agent 生成结构化 `page spec`
- 页面由组件树渲染
- 旁白和讲解文字是附属层
- thinking/tool 只是短暂 ghost telemetry

例子：

Agent 不返回“半导体行业解读.html”，而返回：

- 页面类型：`industry_f10`
- 组件列表：产业链图、核心公司卡片墙、近期催化时间线、风险点、资金热度图
- 每个组件的来源、时间戳、点击动作、置信度

## 五、运行时需要的 primitives

为了把客户端从“页面集合”改成“runtime”，需要先定义一组基础原语。

建议的第一批 primitives：

1. `workspace`
   可保存、可恢复、可继续追问的工作台。

2. `widget`
   图表、卡片、列表、时间线、关系图、讲解层、语音旁白层等可视化单元。

3. `tool`
   选股、画线、比较、摘要、识图、资讯聚合、行业链生成等结构化能力。

4. `memory`
   用户偏好、历史行为、保存结果、常用模板等长期记忆。

5. `subscription`
   对股票、主题、条件的持续关注和后台运行任务。

6. `artifact`
   一次分析或一个页面的结果对象，可保存、复用、分享、迭代。

7. `event bus`
   行情事件、公告事件、用户操作、Agent 输出统一走事件流。

8. `policy`
   权限、延迟、数据来源、是否允许后台运行、是否需要用户确认等约束。

## 六、最终形成的产品交互范式

这类产品不应以“问一句答一句”为基本单位，而应该有三种核心交互范式：

### 1. Open

打开即有东西。

- 首页由 memory 自动生成
- 夜间任务提前准备内容
- 用户一进来就看到“今天该看什么”

### 2. Inspect

围绕金融对象的附着式解释。

- 进入个股页后自动附着讲解能力
- 进入行业页后自动扩展产业链和关联标的
- 用户不是先发问，而是先浏览对象，Agent 再附着解释

### 3. Compose

用自然语言或交互生成新的页面和工作台。

- “帮我生成一个半导体行业页面”
- “把这几个票做成一个观察列表”
- “保存这个工作台，明天继续看”

所以整体更像：

`对象浏览器 + 页面生成器 + 持续代理台`

而不是：

`聊天机器人`

## 七、几个重要判断

### 1. 重点不是有没有问财

问财可以是某个分析 brain，也可以不是。

关键不是谁来生成答案，而是：

- 客户端有没有 runtime
- Agent 能不能持续运行
- 页面能不能被生成和保存
- 数据、工具、memory、组件是不是统一的

### 2. 重点不是“加 AI”，而是“改交互范式”

传统炒股软件的交互范式是：

- 菜单
- tab
- 页面跳转
- 功能按钮

agent-native runtime 的交互范式是：

- 打开即有上下文
- 围绕对象持续解释
- 用一句话生成页面和任务
- 保存成果并沉淀成长期工作台

### 3. 真正的竞争点不是大模型，而是运行时层

大模型能力会趋同，但运行时层不会轻易趋同。

真正的壁垒来自：

- 数据接入
- 工具体系
- 用户 memory
- 页面渲染 runtime
- 后台任务与订阅
- 金融场景下的低延迟与风控

## 九、当前最简洁的结论

本轮讨论的最终结论可以压缩成一句话：

`把炒股客户端做成 agent-native runtime，不是给它加一个 AI 功能，而是把它改造成一个围绕股票对象、用户记忆和实时事件，持续生成页面、调用工具、订阅任务和沉淀成果的执行环境。`
